# UI 绑定与自动生成设计文档计划

## 目标

基于 DGame 当前源码，产出一份可供后续框架演进参考的 UI 绑定与自动生成设计文档，说明现状、优缺点、风险、推荐目标架构和迁移边界。

## 阶段

1. [x] 确认用户要的是设计参考文档，而不是立即修改生成器。
2. [x] 阅读 UI 生命周期、UI 模式、命名规则、项目分层及现有框架文档。
3. [x] 核验生成器、绑定容器、Inspector 与运行时生命周期的源码事实。
4. [x] 编写正式设计文档，明确现状与推荐设计的界限。
5. [x] 校验文档结构、链接与工作区变更范围。

## 关键问题

1. 当前机制如何把 Prefab 节点、序列化组件和生成代码关联起来？
2. 为什么一个 GameObject 多组件、UIWidget 根节点和索引顺序会暴露结构性限制？
3. 哪些现有优点应保留，目标设计如何避免运行时反射和隐式降级？

## 决策

| 决策 | 原因 |
| --- | --- |
| 新文档放在 `docs/ui-binding-generation-design.md` | 与现有架构参考和框架设计文档并列，便于后续检索。 |
| 不修改生成器实现 | 用户本轮请求是生成参考设计文档。 |
| 把“当前实现”与“推荐设计”分章 | 避免把讨论方案误写成仓库既有能力。 |

## 错误记录

| 问题 | 处理 |
| --- | --- |
| 根目录 `task_plan.md` 属于 SweetSugar 任务 | 不修改，使用本次文档专用工作记录。 |
| 一次性文档补丁包含未加补丁前缀的代码块列表行 | 文档尚未写入；后续分段创建并让每行显式成为新增内容。 |
