# UI 绑定与自动生成调研记录

## 用户需求

- 生成一份详细的参考设计文档。
- 说明 UI 绑定与自动生成方案的设计动机、优点、缺点和演进方向。

## 已确认事实

- `UIWindow` 与 `UIWidget` 都经由 `ScriptGenerator -> BindMemberProperty -> RegisterEvent -> OnCreate -> OnRefresh` 生命周期初始化。
- 当前编辑器生成器按节点名称前缀识别一个主要组件，并把引用按扫描顺序写入 `UIBindComponent.m_components`。
- `*_Gen.g.cs` 通过同样的索引与泛型类型读取这些组件，并为 Button、Toggle、Slider 生成监听和 partial 回调声明。
- `m_item` 是 Widget 子树跳过标记；父 Window 不会扫描其内部节点。
- 当前 `UIBindComponent` Inspector 是可重排、可增删、可替换的 `List<Component>`；该列表顺序直接影响运行时绑定语义。
- `GenerateUIComponentScript` 与 `GenerateCSharpScript` 是独立入口：前者更新 Prefab 的组件列表，后者生成或覆盖 `*_Gen.g.cs`。
- 两个入口都从根节点的子节点开始递归，并在遇到 `m_item` 时停止向下扫描，因此 Widget 根节点本身的 Button、Image 等组件不会被自动发现。
- 组件收集发现缺失目标组件时记录错误并跳过该条目；代码生成仍为该节点生成字段并增加索引，可能使之后的索引全部错位。
- `UIWidget.CreateImp` 会将 Widget 加入父 UI，并按 `ScriptGenerator -> BindMemberProperty -> RegisterEvent -> OnCreate -> OnRefresh` 初始化。

## 设计判断

| 判断 | 依据 |
| --- | --- |
| 当前机制属于编辑期 View 引用生成，不是 MVVM 数据绑定 | 数据刷新仍由 `BindMemberProperty`、`OnRefresh` 和业务代码显式处理。 |
| “一个节点名对应一个组件”不能表示 Unity 的多组件对象 | 根节点既可为 Widget 边界，也可同时承载 Button、Image、CanvasGroup、LayoutElement 等。 |
| 目标设计应以显式 BindingEntry 为事实来源 | 组件引用、字段名、稳定 ID 和预期类型应一起持久化，生成器从同一份清单产出 Prefab 数据与 C# 代码。 |

## 补充结论

- 已核验：生成实现类时，若业务 partial 文件已存在，生成器仅记录警告并跳过；新增 Button、Toggle、Slider 时不会合并新的回调骨架。
- 已核验：生成器范围内没有 `BindingId`、checksum 或 duplicate 相关实现；现有 `Validate` 方法仅控制菜单可用性。
