# Deepening

如何在给定依赖的情况下安全地加深一组浅层模块。假设已了解 [LANGUAGE.md](LANGUAGE.md) 中的词汇：**module（模块）**、**interface（接口）**、**seam（接缝）**、**adapter（适配器）**。

DGame 是热更手游工程，没有普遍铺开的单元测试套件。因此"接缝处可测试"这个收益只在逻辑能下沉到 `GameBattle` 纯逻辑域时才真正成立；其余情况下，加深追求的是**分层局部性**、**收口**和**AI 可导航性**，而非"新增一个测试点"。分类依赖时对号入座，判断加深后模块能落在哪一层、以什么方式验证。

## 依赖分类

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O、不碰 Unity。DGame 中：数值/战斗结算、状态判定、红点聚合的纯规则部分。**这是 DGame 唯一天然可单元测试的类别**。若这类逻辑当前缠在 `GameLogic`（依赖 Unity/UI/资源）里，加深的核心动作就是把它下沉到 `GameBattle`（`GameBattle` 只允许纯逻辑，不得依赖引擎/UI/资源/表现），从而既得局部性又得可测性。合并模块并直接通过新接口验证，无需适配器。

### 2. 本地可替代（Local-substitutable）

拥有本地替身的依赖。DGame 中：Luban 配置表（可用内存构造的配置数据替身）、对象池（`GameModule.GameObjectPool`）、本地化文本表（`LocalizationModule`）。如果替身存在则可加深，加深模块用替身运行。接缝是内部的；模块外部接口处不必暴露端口。典型场景：把散落的 `TbXXX` / `ConfigSystem.Instance.Tables` 访问收口进一个 `ConfigMgr` 深模块，业务只面向 `ConfigMgr` 接口。

### 3. 远程但自有 / 通过门面可替换（Ports & Adapters）

跨进程边界的自有服务，或 DGame 中通过 `GameModule` 暴露、实现可被替换的运行时能力（`ResourceModule` 资源加载、`SceneModule` 场景、`AudioModule`、网络请求、存档读写）。在接缝处定义**端口**（接口）：深模块拥有逻辑，具体能力作为**适配器**注入。生产用基于 `GameModule.XXX` 的适配器，需要脱离引擎验证时用内存适配器。

建议形式："在接缝处定义端口，生产实现基于 `GameModule.ResourceModule` 的适配器，验证用内存适配器，逻辑集中在一个深模块里，资源加载/卸载在该模块内成对收口。"

注意：DGame 里更常见的问题不是"缺端口"，而是**资源/配置逻辑跨接缝泄漏**到 UI 或多个模块。多数情况下先做收口（把逻辑归拢到一个深模块，仍直接用 `GameModule`），只有当确有第二个适配器需求（如可测性）时才引入端口，见接缝纪律。

### 4. 真正外部（Mock）

你无法控制的第三方服务（渠道 SDK、支付、推送、第三方登录、埋点）。加深模块把外部依赖作为注入端口接收；验证提供 mock 适配器。DGame 中这类通常已在主工程层，加深时注意不要让 HotFix 编译期写死引用。

## 接缝纪律

- **一个适配器意味着假想的接缝。两个适配器意味着真正的接缝。** 除非至少两个适配器被证明合理（通常是生产 + 验证/测试），否则不要引入端口。单适配器接缝只是间接层。在 DGame，宁可先做收口，别急着为一个门面能力包一层没人替换的接口。
- **内部接缝 vs 外部接缝。** 深模块可以有内部接缝（对实现私有），也有其接口处的外部接缝。不要仅因内部需要就通过接口暴露它们。
- **尊重 DGame 分层与红线。** 接缝不得跨越四层单向依赖反向依赖：不得让 `DGame.Runtime` / `DGame.AOT` 编译期引用 HotFix 业务类型，不得让 `GameBattle` 依赖 `GameLogic`。加深模块的落位遵循 `project-map.md` 与 `client-architecture-codex.md`，访问底层能力仍走 `GameModule`。

## 验证策略：替换，而非叠加

- 能下沉到 `GameBattle` 的纯逻辑：在加深模块接口处编写新单元测试，**接口就是测试面**；原来缠在 UI/模块里的旧测试（若有）作废，删除。
- 不能下沉、仅做收口的模块：DGame 通常没有对应单元测试，"验证"意味着通过项目现有验证方式或在真实启动链路（`GameStart` -> `UIModule.ShowWindow`）里驱动受影响流程，观察行为。收口本身的收益是局部性与可导航性，不是新增测试点，如实说明，别硬安一个测试收益。
- 无论哪种，验证都通过接口断言可观察结果，而非内部状态；描述行为而非实现，使其在内部重构后存活。
