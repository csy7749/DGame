# 接口设计

当用户想为选定的加深候选探索替代接口时，使用此模式。基于 "Design It Twice"（Ousterhout）：第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇：**module（模块）**、**interface（接口）**、**seam（接缝）**、**adapter（适配器）**、**leverage（杠杆）**。

## 流程

### 1. 框定问题空间

在生成替代接口前，为选定的候选写一段面向用户的问题空间说明：

- 任何新接口需要满足的约束（含 DGame 分层与 `GameModule.XXX` 访问约定）。
- 它将依赖的依赖，以及它们属于哪个类别（参见 [DEEPENING.md](DEEPENING.md)）。
- 一个粗略的示意性代码草图，让约束具体化。它不是提案，只是让约束可见。

将此展示给用户，然后进入步骤 2。若用户明确允许并行子代理/多 agent 工作，可让子代理并行产出不同接口；否则由主 agent 本地给出 2-4 个根本不同的接口设计。

### 2. 生成多个接口方向

至少产出 3 个根本不同的接口方向。每个方向使用不同设计约束：

- **最小接口**：最多 1-3 个入口点，最大化每个入口点的杠杆。
- **灵活接口**：支持多种用例和扩展。
- **常见调用者优先**：让默认情况最简单。
- **端口与适配器**（如适用）：围绕跨接缝依赖设计。

如果使用 Codex 子代理，给每个子代理独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝后面是什么），并给每个代理不同的设计约束。不要把期望答案泄漏给子代理。

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 `dgame-dev` skill references 的领域词汇（模块名、事件组、节点前缀、资源路径约定），使命名与架构语言和项目领域语言保持一致。

每个接口方向输出：

1. 接口（类型、方法、参数，以及不变量、顺序、错误模式）。
2. 使用示例，展示调用者如何使用它。
3. 实现在接缝后面隐藏了什么。
4. 依赖策略和适配器（参见 [DEEPENING.md](DEEPENING.md)）。
5. 权衡：杠杆高的地方，杠杆薄的地方。

### 3. 呈现和比较

按顺序呈现设计，让用户可以消化每一个，然后用文字进行比较。按**深度**（接口处的杠杆）、**局部性**（变更集中的位置）和**接缝放置**进行对比。

比较后，给出你的建议：哪个设计最强，为什么。如果不同设计的元素可以组合，提出混合方案。要有主见，用户想要的是判断，不是菜单。
